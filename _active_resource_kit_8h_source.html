00047  * @mainpage Active Resource Kit
00048  *
00049  * What can you do with Active Resource Kit?  Active Resource Kit is yet-another
00050  * RESTful framework. There are others. But Active Resource Kit has a number of
00051  * distinct features.
00052  * 1. It mirrors the Rails Active Resource gem closely. The interface and
00053  * implementation remain as faithful as an Objective-C implementation can
00054  * reasonably be to the Ruby-based originals.
00055  * 2. It offers a very high-level interface to RESTful resources using Core
00056  * Data. You can access remote resources just as if they were in a local Core
00057  * Data store. The implementation uses the new Core Data @c NSIncrementalStore
00058  * API to merge the two dissimilar interfaces.
00059  * 3. It only has Foundation and Core Data as underlying dependencies. Although
00060  * it has two immediate dependencies, Active Model Kit and Active Support Kit
00061  * which fall under the same umbrella framework, Apple's Foundation and Core
00062  * Data kits form the only @em external dependencies. The implementation employs
00063  * only the Foundation framework for network access.
00064  * 4. The framework supports various concurrency models when interacting with
00065  * remote resources: these are the same models offered by Apple's Foundation @c
00066  * NSURLConnection class, i.e. delegated URL connections, synchronous loading or
00067  * queued loading. You can configure according to your requirements on a
00068  * resource-by-resource basis.
00069  * 5. There are no swizzles or other non-standard Objective-C tricks. The
00070  * framework makes extensive use of C blocks for handling completions for both
00071  * asynchronous and synchronous interfaces; this simply follows the pattern set
00072  * by Apple in their URL connection API.
00073  *
00074  * @section setting_up Setting Up an Active Resource-Based Core Data Stack
00075  *
00076  * This is easy to do. Just follow the usual Core Data-prescribed procedure:
00077  * load the model, load the coordinator with the model, add the store to the
00078  * coordinator, and finally attach the coordinator to the context.
00079  *
00080  * See example below. There are some things to especially notice. First, the
00081  * persistent store type specifies the @c ARIncrementalStore type. The URL
00082  * specifies the remote resource, its protocol, host and path. Protocol can be
00083  * @c http, or @c https if you want secure transport. The path can specify the
00084  * site root, i.e. an empty or single-slash path, or can also include a sub-path
00085  * nested within the remote site. The client-side data model must correspond to
00086  * the remote's RESTful interface. The Active Resource incremental store
00087  * utilises the client-side data model in order to deduce the entities, their
00088  * properties and relationships. Core Data itself requires a valid data model.
00089  *
00090  * @code
00091  *  NSBundle *bundle = [NSBundle bundleForClass:[self class]];
00092  *  NSURL *modelURL = [bundle URLForResource:@"MyCoreDataModel" withExtension:@"momd"];
00093  *  NSManagedObjectModel *model = [[NSManagedObjectModel alloc] initWithContentsOfURL:modelURL];
00094  *  NSPersistentStoreCoordinator *coordinator = [[NSPersistentStoreCoordinator alloc] initWithManagedObjectModel:model];
00095  *
00096  *  NSError *__autoreleasing error = nil;
00097  *  NSPersistentStore *store = [coordinator addPersistentStoreWithType:[ARIncrementalStore storeType]
00098  *                                                       configuration:nil
00099  *                                                                 URL:[NSURL URLWithString:@"http://localhost:3000"]
00100  *                                                             options:nil
00101  *                                                               error:&error];
00102  *  // <-- error handling goes here
00103  *
00104  *  NSManagedObjectContext *context = [[NSManagedObjectContext alloc] initWithConcurrencyType:NSMainQueueConcurrencyType];
00105  *  [context setPersistentStoreCoordinator:coordinator];
00106  *  [self setContext:context];
00107  * @endcode
00108  *
00109  * Note that this excerpt uses Automatic Reference Counting, hence the @c
00110  * __autoreleasing specifier for the error pointer. Notice the blatant lack of
00111  * manual auto-releasing.
00112  *
00113  * @section accessing_resources Accessing Resources
00114  *
00115  * You can then access resources using @em only Core Data.
00116  *
00117  * @code
00118  *  NSError *__autoreleasing error = nil;
00119  *  NSFetchRequest *request = [NSFetchRequest fetchRequestWithEntityName:@"Person"];
00120  *  NSArray *people = [[self context] executeFetchRequest:request error:&error];
00121  *  for (NSManagedObject *person in people)
00122  *  {
00123  *      NSString *name = [person valueForKey:@"name"];
00124  *      NSLog(@"person named %@", name);
00125  *  }
00126  * @endcode
00127  *
00128  * You ask Core Data for the Person entities. The answer is a collection of
00129  * managed object representing each Person. You access attributes on the objects
00130  * using standard Cocoa key-value coding. However, underneath the hood, the
00131  * Active Resource incremental store has enacted a RESTful GET request at
00132  * http://localhost:3000/people.json, decoding and caching the active resources
00133  * at the client side.
